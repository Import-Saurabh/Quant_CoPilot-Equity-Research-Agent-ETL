Pure SQL / rule-based
Features 1, 2, 5, 6, 7, 13 — no AI at all

Seven of your fourteen features need zero AI. This is intentional and correct — these features are about arithmetic and joins, not language understanding.

Change detection (feature 1) — example query pattern

SELECT metric, curr_val, prev_val, (curr_val - prev_val)/prev_val * 100 AS pct_delta FROM screener_quarterly WHERE ticker = 'TCS' ORDER BY period DESC LIMIT 2

Run this for every metric column. Store deltas in a metric_deltas table. No LLM involved.

Data conflict flag (feature 6)

When your annual report extractor writes a value to doc_extracted_metrics, run a comparison job against the corresponding Screener row. If abs(doc_val - screener_val) / screener_val > 0.02 (2% tolerance), insert a conflict record. Display both sources to the analyst. No model needed.

  • Use your existing MySQL. No new infrastructure.
  • These features should ship before any AI feature goes live.
  • Speed: sub-100ms response. AI features will be 2-10 seconds. Lead with SQL.
LLM structured extraction
Features 4, 9 — extraction with a schema, not free generation

This is not RAG. You are sending document text directly to the LLM and asking it to return structured JSON matching a predefined schema. No vector database, no retrieval step, no embedding. The LLM is acting as a parser, not a generator.

Concall extraction schema (feature 4)

Define a Pydantic model:

class ConcallExtract(BaseModel):
  guidance: list[GuidanceItem]
  risks: list[str]
  capex_statements: list[CapexItem]
  hiring_commentary: list[str]

Send the full concall text (most concalls are 15-40 pages, under 32k tokens) with instruction: "Extract only stated facts. Do not infer. If a field is not mentioned, return empty list." Use response_format={"type": "json_object"} or Claude's tool use / structured output mode.

Key rules for extraction prompts

  • Always ask for the source sentence verbatim alongside each extracted item. This gives the analyst the page reference and lets you flag if the extraction was wrong.
  • Run extraction once per document, store the JSON in MySQL. Do not re-run on every query — extraction is expensive and deterministic.
  • For annual reports (200+ pages), chunk by section first (MDA, Risk Factors, Financials), extract each section separately, then merge.

Models to use: claude-sonnet-4-20250514 for accuracy, claude-haiku-4-5-20251001 for speed/cost on bulk runs. Do not use GPT-3.5 for financial extraction — it hallucinates numbers.

Document RAG (single-doc retrieval)
Feature 3 — find the right page, no generation

Feature 3 (PDF section navigator) is retrieval only — no generation. Analyst types a search term; system returns the most relevant pages with surrounding context. The analyst reads the actual source. The AI does not paraphrase anything.

Chunking strategy for financial PDFs

  • Annual reports: Chunk by heading/section first (use pdfplumber + regex for "MANAGEMENT DISCUSSION", "RISK FACTORS", etc). Then split sections into 400-600 token chunks with 100-token overlap. Store (ticker, doc_type, year, section_name, page_start, chunk_text) in your vector store.
  • Concall transcripts: Chunk by speaker turn. Each Q&A exchange is a natural chunk. Tag each chunk with speaker_type: management | analyst.

Use hybrid search — not just embeddings

For financial text, BM25 keyword matching often beats pure embedding search for exact terms (ticker names, product names, specific metric names like "EBITDA margin"). Use hybrid retrieval: run both BM25 and embedding search, merge results with Reciprocal Rank Fusion (RRF), return top 5 chunks with page numbers.

Implementation: pgvector for embeddings (add to PostgreSQL, or switch from MySQL for the vector store only) + rank_bm25 Python library. Or use Qdrant which has hybrid search built in.

Embedding model: text-embedding-3-small (OpenAI) or BAAI/bge-small-en-v1.5 (free, runs locally in Docker). For your scale (NIFTY 500 × ~5 years × ~2 doc types), the total corpus is manageable — under 5GB of embeddings.

Multi-document RAG
Feature 8 — cross-doc retrieval + LLM synthesis with citations

Feature 8 (evidence finder) is where classic RAG lives. The analyst poses a thesis claim; the system retrieves supporting and contradicting passages across all documents for that ticker, and the LLM synthesizes a response that cites specific pages. This is the most technically complex v2 feature.

RAG pipeline for evidence finder

  1. Query: Analyst types "management expects margin expansion."
  2. Retrieve: Hybrid search across all docs for this ticker. Filter by doc_type IN ('annual_report', 'concall'). Return top 12 chunks.
  3. Rerank: Pass the 12 chunks through a cross-encoder reranker (cross-encoder/ms-marco-MiniLM-L-6-v2 — runs locally, fast). Rerankers dramatically improve precision over raw embedding similarity for financial text. Keep top 5.
  4. Generate: LLM gets the 5 chunks + analyst query. Instruction: "Identify which chunks support this claim and which contradict it. Quote only from the provided text. Return source (doc, year, page) for every statement."
  5. Display: Show supporting evidence and contradicting evidence separately. Every sentence links to its source chunk and page.

Metadata filtering is critical

Before embedding search, always pre-filter on structured metadata: ticker = 'TCS', year >= 2022, doc_type = 'annual_report'. This reduces search space from 500k chunks to ~2k and dramatically improves precision. Never run a RAG query against the entire corpus without metadata pre-filtering.

Agentic pipeline
Features 10, 12 — orchestrated multi-step LLM calls — v2/v3 only

An agent is only justified when the task requires dynamic decision-making between steps — the output of step N determines what step N+1 does. Most of your features do not need this. Two do.

Feature 10: Sector intelligence (agentic)

The flow: (1) fetch all concall extractions for a sector + quarter from MySQL, (2) LLM clusters themes by topic (returns JSON groupings), (3) for each cluster with 3+ companies mentioning it, LLM generates a sector-level insight with company names, (4) store in sector_themes table. This requires dynamic branching because step 3 only runs for clusters that cross the threshold. A simple sequential pipeline won't handle that cleanly.

Framework: Plain Python is fine. You don't need LangChain or LangGraph for this — those frameworks add abstraction overhead that makes debugging harder. Write the orchestration logic yourself in FastAPI background tasks.

Feature 12: Draft assistant (agentic generation)

Step 1: Analyst selects 3-5 saved insights from their workspace. Step 2: Agent retrieves the source passages for each insight via RAG. Step 3: LLM generates a draft paragraph where every claim cites a specific retrieved passage. Step 4: Compliance watermark appended automatically.

This is the only feature with genuine LLM generation. Keep it to paragraph level — do not generate full-section reports.

Important: Do not use agents for features 1-9. Agents are slower (3-10x latency), more expensive, and harder to debug. A deterministic pipeline that does the same thing every time is better for a research tool than an agent that might decide to do something different.

What not to use

LLM fine-tuning — skip entirely for now

Fine-tuning is the right tool when a base model consistently fails at a specific task AND you have 500+ labeled examples AND you've exhausted prompting improvements. You have none of these conditions yet.

  • For extraction (feature 4): Structured output + good prompting beats fine-tuning. Claude and GPT-4o already understand financial language including Indian company terminology. A well-designed schema + few-shot examples in the prompt will get you to 85-90% accuracy.
  • For RAG (features 8, 10): Better chunking and reranking improves RAG quality more than fine-tuning the generator. Invest in retrieval quality first.
  • When to revisit: After 6 months of production use, if you have labeled extraction errors across 1000+ documents and a specific failure pattern (e.g., Hindi-language sections misparse), then consider fine-tuning a small model (Llama 3 8B) specifically for that extraction task.

Full agent frameworks (LangChain, AutoGPT, CrewAI) — avoid for v1

These frameworks obscure what your code is actually doing, make debugging hard, and add latency. For a financial research tool where every output must be traceable and correct, you want maximum control over each step. Write the orchestration yourself in Python. Use the Anthropic SDK or OpenAI SDK directly. Add frameworks only when plain Python becomes genuinely unmanageable.
